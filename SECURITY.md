# 安全策略

SQL Studio 会处理数据库访问地址、登录会话和可选的本地保存凭据。请勿在公开 Issue、Discussion、PR、截图或日志中披露真实凭据、Cookie、Session、内部域名、数据库内容或可识别个人的信息。

## 支持范围

| 版本 | 安全更新 |
|------|----------|
| `main` / 最新发布版本 | 支持 |
| 更早版本 | 不保证 |

## 私密报告漏洞

请优先使用代码托管平台提供的私密漏洞报告功能（GitHub 仓库中的 **Security → Report a vulnerability**）。如果当前镜像未启用该功能，请通过维护者在托管平台公开的私密联系方式报告，不要创建公开 Issue。

报告建议包含：

- 受影响的客户端、版本和操作系统；
- 最小复现步骤与影响说明；
- 已脱敏的日志或 PoC；
- 是否已在其他渠道披露。

维护者目标是在 3 个工作日内确认收到、7 个工作日内给出初步分级，并在修复可用后协调披露时间。

## 当前安全边界

- 数据库权限、SQL 审核、脱敏和查询行数限制仍由 Archery 服务端控制；客户端不会绕过服务端策略。
- 密码存入当前 Windows 用户的凭据管理器，其他配置与查询历史位于应用配置目录。
- Rust HTTP 客户端为每个环境维护独立 Cookie Jar，并直接构造 Django CSRF 请求头；会话不会在环境间共享。
- 客户端当前强制绕过系统代理并接受无效/自签 TLS 证书，只适合受控内网环境；连接公网服务前应评估中间人攻击风险。
- 当前 Tauri 前端仅加载打包的本地资源，但桌面配置尚未启用 CSP。任何可导致前端脚本注入的问题都应按高风险报告。

更多数据处理说明见 [PRIVACY.md](PRIVACY.md)。
