# 流水线与通知中心执行计划

## 1. 实施目标

按照 [技术方案](technical-design.md) 分阶段交付流水线和软件通知中心。每个阶段必须通过验证门禁后才能进入下一阶段，避免同时改造存储、调度、执行器和界面导致问题难以定位。

## 2. 阶段总览

| 阶段 | 目标 | 可交付结果 |
|------|------|------------|
| 0 | 冻结语义与准备测试基线 | 关键产品规则明确、现有测试基线通过 |
| 1 | 建立 SQLite 与共享服务基础 | `workflow.db`、迁移、Archery 服务拆分 |
| 2 | 实现流程管理与发布 | 草稿、节点编辑、校验、不可变版本 |
| 3 | 实现手动 SQL 流水线 | 手动执行、记录、产物、历史详情 |
| 4 | 接入消息构建与钉钉插件 | 标准产物链路与终止节点 |
| 5 | 实现软件通知中心 | 未读角标、消息抽屉、Windows 通知 |
| 6 | 实现后台定时调度 | Cron、时区、串行队列、启动恢复 |
| 7 | 稳定性、安全与发布准备 | 回归、安全审计、迁移与构建验证 |
| 8 | 后续扩展 | Python 脚本资源与版本 |

## 3. 阶段 0：语义冻结与基线

### 工作项

1. 确认首期每个流程只允许一条业务 SQL；
2. 确认关机或程序未运行期间错过的 Cron 不自动补跑；
3. 确认执行产物和通知的默认保留策略；
4. 明确手动执行是否允许临时使用当前会话，定时执行必须使用已保存凭据；
5. 记录现有 `npm test`、Rust 测试、格式检查和构建结果。

### 完成门禁

- 产品语义写入需求说明；
- 当前主分支测试基线可复现；
- 不修改 `store.json` schema 的约束得到确认。

## 4. 阶段 1：存储与宿主基础

### 工作项

1. 引入 SQLite、CBOR、UUID 和时间相关依赖；
2. 建立 `workflow.db` 初始化与事务化迁移；
3. 实现连接配置、外键、WAL 和 busy timeout；
4. 将现有 Archery HTTP、登录、Cookie 和查询能力抽成共享服务；
5. 将现有 JSON KV 与凭据逻辑移入独立存储模块，但保持外部行为和数据格式不变；
6. 增加单实例能力；
7. 建立仓储接口、时钟接口和可注入测试实现。

### 完成门禁

- 新安装和已有配置目录均可初始化数据库；
- 迁移失败显式报错且不回退 JSON；
- 现有登录、资源树、SQL 控制台和钉钉测试功能行为不变；
- 会话按环境 ID 和用户身份隔离；
- 后端测试单次运行不超过 60 秒。

## 5. 阶段 2：流程管理与发布

### 工作项

1. 实现流程定义、草稿节点、版本和版本节点仓储；
2. 实现流程创建、编辑、复制、软删除、启停；
3. 实现环境、实例、数据库、PostgreSQL 模式和 SQL 配置；
4. 建立插件注册表和插件资源查询接口；
5. 实现节点添加、删除、排序和终止节点限制；
6. 实现后端发布校验与事务化版本快照；
7. 建立流程列表和编辑一级页面。

### 完成门禁

- PostgreSQL 缺少模式时不能发布；
- SQL 为空、类型不兼容或终止节点位置错误时不能发布；
- 已发布版本不能被原地修改；
- 删除或修改草稿不破坏历史版本；
- 前端限制和 Rust 后端校验均有测试。

## 6. 阶段 3：手动 SQL 流水线

### 工作项

1. 实现执行、节点执行和产物仓储；
2. 实现手动创建 `pending` 执行；
3. 实现任务原子领取和同流程串行执行；
4. 通过共享 Archery 服务执行 SQL；
5. 将查询结果编码为 CBOR `table`，非查询结果保存为 `object`；
6. 实现失败停止、跳过后续节点和脱敏错误记录；
7. 实现执行历史、执行详情和表格产物查看。

### 完成门禁

- 手动执行产生完整总体记录和 SQL 节点记录；
- SQL 成功、失败、凭据失效均有明确结果；
- CBOR 编解码能保留数字、布尔、空值、数组和对象类型；
- 应用异常退出后遗留执行可标记为 `interrupted`；
- SQLite、日志和前端事件中不存在明文凭据。

## 7. 阶段 4：插件链路

### 工作项

1. 定义 `PluginExecutor` 接口与运行上下文；
2. 实现输入输出类型运行时校验；
3. 实现消息构建插件的文本、Markdown、模板、循环和预览；
4. 将现有钉钉配置适配为 `dingtalk:default` 插件资源；
5. 支持流程选择具体插件资源；
6. 实现 `dispatching` 状态和外部调用摘要；
7. 支持查看消息产物预览和钉钉发送摘要。

### 完成门禁

- `SQL → 消息构建 → 钉钉` 可完整执行；
- 插件收到的输入是前一节点保存的标准产物；
- 输出类型不匹配时立即失败；
- 添加钉钉后不能继续添加节点；
- 钉钉发送后崩溃不会自动重发，历史明确显示“发送结果未知”；
- Webhook 和 Secret 仍只存在于 Windows 凭据管理器。

## 8. 阶段 5：软件通知中心

### 工作项

1. 实现 `app_messages`、`message_deliveries` 和偏好仓储；
2. 将流程最终状态与应用消息放在同一事务提交；
3. 实现失败、中断、调度异常、凭据失效和发送失败消息映射；
4. 增加顶栏铃铛和持续未读角标；
5. 实现通知抽屉、筛选、单条已读、全部已读和详情跳转；
6. 接入 Windows 原生通知；
7. 更新托盘 tooltip，点击通知时恢复窗口并定位消息；
8. 实现统一通知脱敏器和去重键。

### 完成门禁

- 新消息产生后即使主窗口已关闭、WebView 已销毁也能明确提醒；
- 前端重载或漏掉事件后，未读数量可从 SQLite 恢复；
- 同一执行终态不会重复产生消息；
- 打开通知中心不会自动全部已读；
- 原生通知失败被记录，但不改变流水线状态；
- 通知内容不包含 SQL、查询结果、凭据或未清洗响应。

## 9. 阶段 6：定时调度

### 工作项

1. 实现 Cron 和时区解析、校验及 `next_run_at` 计算；
2. 实现后台调度任务和主动唤醒机制；
3. 在同一事务创建执行记录并推进下一次计划；
4. 使用唯一约束防止重复触发；
5. 实现同流程手动与定时任务统一排队；
6. 会话失效时从凭据管理器重新登录；
7. 实现启动恢复、错过计划记录和通知；
8. 增加时区、夏令时和并发竞争测试。

### 完成门禁

- 同一计划时间只生成一条执行记录；
- 同一流程不会并行执行；
- 主窗口关闭并销毁 WebView 后定时任务继续运行；
- 无凭据或凭据失效时明确失败并通知；
- 错过计划不会静默补跑或集中补跑；
- 两个调度领取者竞争时仍不会重复执行。

## 10. 阶段 7：稳定性与发布准备

### 工作项

1. 覆盖数据库损坏、磁盘写入失败、应用崩溃和网络超时；
2. 检查产物大小、查询分页和历史查看性能；
3. 完成凭据、日志、错误和通知内容安全审计；
4. 验证旧版 `store.json` 和钉钉默认凭据兼容；
5. 完成 Windows 安装、升级、托盘和系统通知验证；
6. 更新架构、配置、测试和用户使用文档。

### 完成门禁

- `npm test` 通过；
- `cargo test` 在 60 秒超时内通过；
- `cargo fmt --check` 通过；
- `git diff --check` 通过；
- Windows 构建与关键 E2E 通过；
- 安全检查确认 SQLite、JSON、日志和通知中不存在明文凭据。

## 11. 阶段 8：Python 插件扩展

Python 插件不阻塞首期流水线上线，在核心链路稳定后实施。

### 工作项

1. 实现脚本定义和不可变脚本版本；
2. 流程节点引用明确脚本版本；
3. 定义受控输入、输出和非敏感运行上下文；
4. 校验脚本返回值和声明输出类型；
5. 实现超时、异常、资源限制和安全隔离；
6. 实现脚本管理、版本发布和执行结果查看。

### 完成门禁

- 脚本不能读取数据库或其他插件凭据；
- 脚本异常和输出不合法时节点明确失败；
- 历史执行始终可定位到原脚本版本；
- 不以回传原始输入作为失败降级路径。

## 12. 测试矩阵

| 层级 | 重点 |
|------|------|
| 领域单元测试 | 状态机、节点类型、终止节点、Cron、去重键 |
| 仓储测试 | 迁移、事务、外键、唯一约束、不可变版本 |
| 执行器测试 | SQL、产物串联、失败停止、`dispatching`、启动恢复 |
| 安全测试 | 凭据不落库、不入日志、不进通知 |
| 前端单元测试 | 节点排序、发布错误、未读状态、详情跳转 |
| 集成测试 | 手动执行、定时执行、消息落库、Windows 通知调用 |
| 回归测试 | 控制台、环境、历史、资源树、插件管理 |

所有后端自动化测试必须设置不超过 60 秒的硬超时。

## 13. 交付与提交策略

- 每个阶段独立提交，避免数据库、调度、界面和插件混在一个提交中；
- 数据库迁移与对应仓储代码必须同一提交；
- 功能代码、测试和相关文档同步提交；
- 不提交明文凭据、真实 Webhook、测试数据库密码或包含业务结果的测试产物；
- 每个阶段通过完成门禁后再合并或进入下一阶段。
