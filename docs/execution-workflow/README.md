# 执行流程需求说明

配套文档：

- [流水线与通知中心技术方案](technical-design.md)
- [流水线与通知中心执行计划](implementation-plan.md)

## 1. 背景与目标

SQL Studio 需要在现有交互式 SQL 控制台之外，新增可定时运行的“执行流程”。执行流程以 SQL 查询为固定起点，按用户配置的顺序调用数据加工与消息推送插件，并完整记录每次运行及各步骤结果。

首期目标：

- 支持手动执行和定时执行 SQL；
- 执行前选择环境、实例、数据库，PostgreSQL 还必须选择模式；
- 保存流程执行记录、SQL 结果、插件输入输出和错误，支持历史查看；
- 支持按顺序添加已配置的插件；
- 插件输出作为下一个插件的输入；
- 推送类终止插件必须是最后一步；
- 首个终止插件为已实现的钉钉机器人消息；
- 后续可扩展 Python 脚本、消息构建、飞书、企业微信和邮件等插件。

## 2. 范围边界

### 2.1 本期范围

本期采用单向、线性的执行流水线：

```text
触发器 → 数据源 → SQL → 加工插件 0..N 个 → 终止插件 0..1 个
```

示例：

```text
每天 09:00
  → 生产环境 / PostgreSQL / order_db / public
  → 订单统计 SQL
  → Python 脚本“生成日报数据”
  → 消息构建“销售日报模板”
  → 钉钉机器人“运营群机器人”
```

### 2.2 非本期范围

- 不支持 DAG、条件分支、循环或并行节点；
- 不迁移或重构现有环境、查询历史、控制台会话的 JSON 存储；
- 不允许插件直接调用其他插件；
- 不在流程定义、执行记录或日志中保存数据库密码、Webhook、访问令牌和签名密钥；
- 不为失败节点提供静默跳过、模拟成功或隐式降级。

### 2.3 首期已确认规则

- 每个流程只允许配置一条业务 SQL，避免多 SQL 的部分成功和多结果产物语义不明确；
- 手动执行可以复用当前有效的 Archery 会话；定时执行必须具备 Windows 凭据管理器中的已保存凭据；
- 程序未运行、系统关机或休眠期间错过的 Cron 首期不自动补跑，恢复后等待下一次计划并记录提醒；
- 执行记录、数据产物和应用通知首期不自动清理，后续通过明确的保留策略增加清理能力；
- 外部推送发送结果不确定时标记为 `interrupted`，不得自动重发。

## 3. 核心概念

| 概念 | 说明 |
|------|------|
| 流程定义 | 用户编辑的流程草稿，包括触发器、数据源、SQL 和节点顺序 |
| 流程版本 | 发布时生成的不可变快照，保证历史执行可复现 |
| 插件定义 | 插件的能力描述，包括输入类型、输出类型、是否为终止节点 |
| 插件资源 | 插件管理中配置的具体资源，例如某个钉钉机器人或某条 Python 脚本 |
| 流程节点 | 流程对插件资源及其版本的引用，并包含本次流程使用的节点参数 |
| 流程执行 | 一次手动或定时运行的总体记录 |
| 节点执行 | SQL 或某个插件节点的一次执行记录 |
| 数据产物 | 节点产生的结构化数据、文本、消息或文件引用 |

## 4. 功能需求

### 4.1 流程管理

- 支持创建、编辑、复制、启用、停用和删除流程；
- 流程至少包含名称、触发器、数据源和 SQL；
- 流程草稿可修改，启用前必须发布为不可变版本；
- 修改已启用流程后需要重新发布，新执行使用新版本，历史执行保留原版本；
- 删除流程不得级联删除历史执行记录，历史记录使用流程名称和版本快照展示。

### 4.2 数据源选择

- 环境、实例和数据库为必选项；
- PostgreSQL 模式为必选项，其他数据库类型不显示模式选择；
- 数据源选择继续复用现有环境和 Archery 实例能力，不复制数据库密码；
- 流程中保存环境 ID、实例标识、数据库、模式和数据库类型的引用或非敏感快照；
- 执行时若引用的环境、实例或凭据已失效，流程明确失败并记录原因。

### 4.3 触发方式

- 支持手动立即执行；
- 支持 Cron 定时执行；
- 定时配置必须包含时区，默认使用应用配置的时区；
- 同一次触发只创建一个流程执行记录；
- 调度成功不等于执行成功，两者状态需要分别记录；
- 首期同一流程按单实例串行执行，避免前一次未结束时产生相互覆盖的数据状态。

### 4.4 SQL 节点

- SQL 是流程的固定首个业务节点，不能删除或调整顺序；
- 首期创建流程时必须明确 SQL 类型为“查询”或“命令”；查询声明输出 `table`，命令声明输出 `object`，运行时结果必须与声明一致；
- SQL 执行沿用现有 Archery 请求链路与权限边界；
- 记录开始时间、结束时间、耗时、状态、影响行数、结果产物和错误；
- SQL 失败后立即终止流程，后续插件不执行；
- SQL 查询结果输出为 `table`，非查询语句输出为包含影响行数的 `object`。

### 4.5 插件选择与排序

- 只能选择插件管理中已配置且可用的插件资源；
- 支持添加、删除和调整非终止插件顺序；
- 节点顺序就是运行顺序；
- 添加节点时检查前一节点输出类型是否被新节点接受；
- 终止插件只能位于最后，选中后禁用继续添加节点；
- 删除终止插件后恢复添加节点能力；
- 发布流程时再次执行完整校验，不能仅依赖前端限制。

### 4.6 Python 脚本插件

Python 插件是执行器，具体脚本作为可管理资源独立存在。

- 支持创建、编辑、复制、启用、停用和删除多条脚本；
- 流程选择 Python 插件时必须指定脚本及其版本；
- 脚本版本发布后不可变，流程版本引用明确的脚本版本；
- 脚本使用统一入口接收执行上下文并返回标准插件结果；
- 脚本需要声明本次节点的预期输出类型，以便发布前校验数据链路；
- 脚本异常、返回值不合法或输出类型不匹配时节点失败，不继续传递原始数据；
- Python 运行时不得自动获得数据库密码、钉钉密钥或其他插件凭据。

### 4.7 消息构建功能

消息构建是普通加工插件，不是推送插件，负责把 `table`、`object` 或 `text` 转换为标准 `message`。

- 支持普通文本和 Markdown；
- 支持标题、正文模板、流程变量和数据字段引用；
- 支持表格行循环渲染；
- 支持消息预览；
- 支持配置查询结果为空时是否生成消息；
- 可输出 `@所有人` 或指定人员等通用提及信息；
- 不包含钉钉 Webhook、签名或网络发送逻辑。

### 4.8 钉钉机器人插件

- 钉钉机器人属于 `sink` 类型的终止插件；
- 消费标准 `message`，也可明确支持 `text`；
- 只负责读取凭据、转换钉钉请求、加签、发送和解析响应；
- 发送成功后输出 `none`，记录非敏感响应码、消息类型和执行摘要；
- Webhook、`access_token` 和加签密钥继续保存于 Windows 凭据管理器；
- POST 请求不自动重试，避免超时后重复推送。

### 4.9 执行记录与结果查看

执行历史至少支持按流程、状态和时间筛选，并展示：

- 流程名称、流程版本、触发方式和计划时间；
- 总体状态、开始时间、结束时间和总耗时；
- 使用的数据源非敏感快照；
- SQL 与插件节点的顺序、状态和耗时；
- 每个节点的输入产物、输出产物和执行摘要；
- 明确的错误节点、错误信息和发生时间；
- 钉钉等外部推送的非敏感响应结果。

## 5. 插件数据链路

### 5.1 执行原则

插件之间不直接通信。流程执行器读取当前节点的输入产物，调用插件，再把插件输出保存为新产物并交给下一个节点。

```text
SQL 输出产物 A
  → Python(A) 输出产物 B
  → 消息构建(B) 输出产物 C
  → 钉钉(C) 输出发送摘要
```

每个节点必须在事务记录中明确关联 `input_artifact_id` 和 `output_artifact_id`，从而还原完整数据链路。

### 5.2 标准数据类型

| 类型 | 内容 | 常见生产者 | 常见消费者 |
|------|------|------------|------------|
| `table` | 字段定义与多行数据 | SQL、Python | Python、消息构建 |
| `object` | 单个结构化对象 | SQL、Python | Python、消息构建 |
| `text` | 普通文本 | Python、消息构建 | 消息构建、推送插件 |
| `message` | 标题、格式、正文和提及信息 | 消息构建、Python | 钉钉、飞书、邮件 |
| `files` | 文件产物引用及元数据 | 导出插件、Python | 上传或推送插件 |
| `none` | 无下游业务数据 | 终止插件 | 无 |

插件定义必须声明 `accepts`、`produces`、`category` 和 `terminal`。流程节点还要声明本次配置的确定输出类型，避免动态脚本导致流程无法提前校验。

### 5.3 统一运行上下文

运行时上下文由执行器构造，包含：

- 执行 ID、流程 ID 和流程版本；
- 触发类型和计划执行时间；
- 数据源的非敏感信息；
- 当前节点及前序节点摘要；
- 当前输入产物；
- 流程变量；
- 受控的运行时能力。

上下文只存在于运行时内存与受控产物存储中，不包含任何明文凭据。

### 5.4 插件返回结果

插件返回结果包含：

- `status`：成功或失败；
- `output_type`：实际输出类型；
- `output`：待保存的数据产物；
- `variables`：需要向后传递的非敏感流程变量；
- `summary`：用于历史记录展示的非敏感摘要；
- `error`：失败时的明确错误信息。

实际输出类型与节点声明不一致时，执行器将节点标记为失败。

## 6. 存储方案

### 6.1 结论

新执行流程使用独立 SQLite 数据库，推荐路径：

```text
%APPDATA%\com.fanxiaofan.sql-studio\workflow.db
```

现有功能继续使用：

```text
%APPDATA%\com.fanxiaofan.sql-studio\store.json
```

本需求不迁移、不拆分、不重写 `store.json`。两个存储由不同仓储模块负责，避免新功能引发已有环境、历史和控制台会话的大范围改动。

### 6.2 为什么选择 SQLite

执行流程天然包含流程、版本、节点、脚本、调度、执行、节点执行和产物之间的关联。与单文件 JSON 相比，SQLite 更适合：

- 事务性创建执行记录和更新节点状态；
- 按流程、状态、时间进行筛选和分页；
- 使用外键维持流程版本、节点和执行记录关系；
- 在应用异常退出后恢复 `running` 状态并明确标记中断；
- 通过索引控制历史记录查询成本；
- 使用 schema migration 演进新功能；
- 单文件部署，不需要额外数据库服务。

### 6.3 数据库存储边界

SQLite 保存：

- 流程定义及不可变版本；
- 调度配置和下次执行时间；
- 插件资源的非敏感配置引用；
- Python 脚本及版本；
- 流程执行和节点执行记录；
- SQL 结果、插件输入输出等数据产物；
- 非敏感错误和外部调用摘要。

SQLite 不保存：

- 环境登录密码；
- 钉钉 Webhook、`access_token` 和签名密钥；
- 其他插件的 API Key、Token 或密码；
- 可由凭据管理器保存的任何敏感认证材料。

敏感配置在 SQLite 中只保存稳定的凭据引用 ID，执行时由 Rust 宿主从 Windows 凭据管理器读取。

### 6.4 建议的数据表

| 表 | 作用 |
|----|------|
| `workflow_definitions` | 流程当前草稿和启停状态 |
| `workflow_versions` | 已发布的不可变流程版本 |
| `workflow_nodes` | 版本内 SQL 与插件节点及顺序 |
| `workflow_schedules` | Cron、时区和下次运行时间 |
| `plugin_resources` | 插件资源的名称、类型、状态和非敏感配置 |
| `script_definitions` | Python 脚本资源 |
| `script_versions` | 不可变脚本内容和版本信息 |
| `workflow_executions` | 每次流程运行的总体状态 |
| `node_executions` | 每个节点的输入、输出、状态和耗时 |
| `execution_artifacts` | SQL 结果和插件数据产物 |
| `schema_migrations` | SQLite schema 版本与迁移记录 |

核心约束：

- 同一流程版本内节点顺序唯一；
- 终止节点只能是最后一个节点；
- 节点执行必须关联流程执行；
- 输入和输出产物必须关联同一次流程执行；
- 流程版本、脚本版本发布后不可原地修改；
- 删除业务配置采用状态标记，不能破坏历史执行引用。

### 6.5 非 JSON 的产物编码

SQLite 的关系字段使用普通列存储；任意结构的 `table`、`object` 和 `message` 产物使用 CBOR 编码后写入 BLOB，不使用 JSON 文件或 JSON 文本作为底层持久化格式。

`execution_artifacts` 至少记录：

- 产物 ID、执行 ID 和生产节点 ID；
- 数据类型；
- 编码格式，例如 `cbor`、`utf8` 或 `binary`；
- BLOB 内容或文件产物引用；
- 行数、字节数、内容摘要和创建时间；
- 是否包含需在界面中脱敏展示的业务数据标记。

文本消息可直接使用 UTF-8 TEXT；结构化表格和对象使用 CBOR，可以保留数字、布尔值、空值、数组和对象类型，并避免依赖 JSON 文本解析。

### 6.6 与现有 JSON 的集成边界

```text
现有 StoreRepository
  └── store.json：环境、查询历史、控制台会话等现有功能

新增 WorkflowRepository
  └── workflow.db：流程、脚本、调度、执行记录、节点与产物

CredentialRepository
  └── Windows 凭据管理器：密码、Webhook、Token、Secret
```

执行流程只通过现有公开接口读取环境信息，不直接修改 `store.json` 的结构。新旧存储失败分别暴露错误，不能在 SQLite 失败时静默回退到 JSON。

## 7. 执行状态与失败处理

流程状态：

```text
pending → running → succeeded
                  ↘ failed
                  ↘ cancelled
                  ↘ interrupted
```

节点状态与流程状态独立记录。任一必需节点失败后：

1. 保存节点错误与已产生的输出；
2. 不执行后续节点；
3. 将流程标记为 `failed`；
4. 已成功节点保持成功状态；
5. 未开始节点保持 `pending` 或标记为 `skipped_due_to_failure`，界面明确展示原因。

应用启动时检查遗留的 `running` 执行，将无法继续的记录标记为 `interrupted`，不伪装为成功，也不自动重发终止插件消息。

## 8. 安全要求

- 数据库密码和插件凭据只由 Rust 宿主读取；
- 前端、SQLite、`store.json`、执行上下文和日志均不得出现明文凭据；
- Python 脚本默认只接收业务数据和非敏感元数据；
- 所有插件输入、脚本返回值和外部响应都必须在边界处校验；
- 流程详情和历史结果按潜在敏感数据处理；
- 错误记录保留定位所需信息，但必须过滤请求头、Webhook 查询参数和密钥；
- 外部推送默认不自动重试，避免消息重复；
- SQLite 使用事务保证执行状态、节点状态和产物引用一致。

## 9. 界面需求

### 9.1 执行流程列表

- 展示流程名称、状态、触发方式、最近执行结果和下次执行时间；
- 提供新建、编辑、启停、手动执行和查看历史入口。

### 9.2 流程编辑页

- 基础信息；
- 定时配置；
- 环境、实例、数据库和 PostgreSQL 模式选择；
- SQL 编辑与校验；
- 按顺序排列的插件节点；
- 添加插件时选择插件资源，例如具体 Python 脚本或钉钉机器人；
- 终止插件添加后隐藏或禁用“继续添加”；
- 展示相邻节点的数据类型连接状态；
- 发布前集中展示所有校验错误。

### 9.3 执行详情页

- 顶部展示总体状态与时间信息；
- 按顺序展示 SQL 和插件节点；
- 节点可展开查看输入、输出、摘要与错误；
- `table` 使用数据网格查看，`message` 使用消息预览查看；
- 敏感字段和凭据引用不得展示原值。

## 10. 验收标准

- 可以创建包含数据源、SQL、Python/消息加工和钉钉终止节点的线性流程；
- PostgreSQL 未选择模式时不能发布；
- 不兼容的插件输入输出类型不能连接或发布；
- 添加终止插件后不能继续添加节点；
- 手动和定时执行均生成完整流程记录及逐节点记录；
- 每个插件收到前一节点的标准产物，顺序与界面配置一致；
- 任一节点失败后后续节点不执行，错误清晰可查；
- 历史页面可查看 SQL 结果、加工结果、消息预览和钉钉发送摘要；
- 新功能数据写入 `workflow.db`，现有环境、历史与控制台会话继续写入 `store.json`；
- 所有敏感凭据仍只保存在 Windows 凭据管理器；
- SQLite 初始化或迁移失败时显式报错，不回退到 JSON 或模拟成功。
