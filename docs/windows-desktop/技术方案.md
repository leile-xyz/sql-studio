# SQL Studio Windows 桌面端 — 技术方案与执行计划

> 基于现有浏览器扩展（`extension/`，约 2100 行 vanilla JS）迁移为 Windows 桌面应用。
> 方案基线于 2026-07-09 与用户对齐确定。
>
> 本文前半部分保留迁移时的历史设计，实际落地差异和当前验证结果见第 7 节；当前产品文档以 [README.md](./README.md) 为准。

## 1. 背景与目标

浏览器扩展版已实现完整功能（多环境登录、库表树浏览、翻页查数据、排序筛选、表结构/DDL、自由 SQL 控制台、CSV 导出、查询历史）。桌面端目标：

- **1:1 平移全部现有功能**，并追加桌面专属特性（原生另存为、窗口状态记忆等）；
- 摆脱浏览器载体：无需开发者模式加载扩展、不受浏览器策略变更影响；
- **密码存储安全升级**：扩展的 AES-GCM 密钥同存本地形同混淆，桌面端改用 Windows 凭据管理器（DPAPI）。

## 2. 已确定的技术决策

| 决策项 | 结论 | 理由 |
|--------|------|------|
| 框架 | **Tauri 2**（Rust 宿主 + WebView2） | 前端代码几乎整体复用；产物 ~5MB；Win10/11 自带 WebView2 Runtime；工作区已有 Rust 项目经验 |
| 网络层 | Rust 端 `reqwest` 发请求，**按环境独立 Cookie Jar** | 桌面 HTTP 客户端无 CORS，可直接设置 `Origin`/`Referer` 通过 Django CSRF 校验；扩展里的 declarativeNetRequest 改写方案整体作废 |
| 密码存储 | **Windows 凭据管理器**（`keyring` crate，底层 DPAPI） | 密钥不落盘，绑定 Windows 用户；防离线拷文件与跨用户读取 |
| 其他配置存储 | JSON 文件，`%APPDATA%\com.fanxiaofan.sql-studio\store.json` | 环境列表、查询历史、窗口状态等非敏感数据 |
| 分发 | **单 exe 便携版**，手动分发更新 | 内网小范围工具，无需安装包与自动更新 |
| 功能范围 | 1:1 平移 + 桌面特性（原生另存为、窗口状态记忆；全局快捷键/Excel 导出为可选后续） | — |

## 3. 架构对比

```
扩展（现状）                                桌面端（目标）
┌────────────────────────┐                ┌────────────────────────┐
│ app.html/js/css (UI)   │                │ index.html/js/css (UI) │  ← 复用并模块化
│   └ lib/api.js         │                │   └ lib/api.js         │  ← sendMessage → invoke()
│     chrome.runtime     │                │     @tauri-apps/api    │
│     .sendMessage       │                ├────────────────────────┤
├────────────────────────┤                │ Rust 宿主 (src-tauri)   │  ← 重写 background.js
│ background.js (SW)     │                │  · reqwest + CookieJar │
│  · fetch + credentials │      =>        │    （每环境一个 Client）│
│  · DNR 改写 Origin      │（整体删除）     │  · 直接设 Origin/Referer│
│  · chrome.cookies 读   │                │  · keyring 存取密码     │
│    csrftoken           │                │  · plugin-store 配置    │
├────────────────────────┤                │  · plugin-dialog/fs    │
│ chrome.storage.local   │                │    （CSV 另存为）       │
│ crypto.js (AES-GCM)    │（删除）         └────────────────────────┘
└────────────────────────┘
```

关键差异：扩展里 `background.js` 的三大职责（绕 CORS、DNR 改写 Origin/Referer、依赖浏览器 Cookie 机制）在桌面端**全部不需要**——`reqwest` 可自由设置任何请求头，Cookie 由自建 Jar 管理。

## 4. 模块改动清单

### 4.1 新建：`desktop/`（Tauri 工程）

```
desktop/
├── src-tauri/
│   ├── Cargo.toml            # tauri 2, reqwest(cookies,json), keyring, serde
│   ├── tauri.conf.json       # 窗口配置、CSP、打包（单 exe）
│   └── src/
│       ├── main.rs / lib.rs  # 入口、command 注册、状态管理
│       ├── http.rs           # 每环境 Client + Cookie Jar；requestJson/登录页检测
│       ├── commands.rs       # 业务 command（见 4.2）
│       └── credential.rs     # keyring 封装（save/get/delete）
└── src/                      # 前端，自 extension/ 平移
    ├── index.html / app.css / app.js
    └── lib/ (api.js, store.js, actions.js)   # crypto.js 删除
```

### 4.2 Rust command 与扩展 handler 对照

| 扩展（background.js） | Rust command | 说明 |
|---|---|---|
| `login` | `login(env_id, origin, username, password)` | GET `/login/` 取 csrftoken → POST `/authenticate/`；csrftoken 从 Jar 读取而非 `chrome.cookies` |
| `SESSION` 探测 | `check_session(env_id, origin)` | 取实例列表能否成功 |
| `getInstances` | `get_instances` | GET `/group/user_all_instances/?tag_codes[]=can_read` |
| `getDatabases/Schemas/Tables/Columns` | `get_resource` | 合并为一个 command，`resource_type=database/schema/table/column` 参数区分；表、列请求透传 `schema_name` |
| `describeTable` | `describe_table` | POST `/instance/describetable/` 并透传 `schema_name`；MySQL DDL 在前端解析，PostgreSQL 按列元数据解析且不展示 DDL 页签 |
| `runQuery` | `run_query` | POST `/query/` |
| —（新增） | `save_credential / get_credential / delete_credential` | keyring：service=`sql-studio`，account=`{envId}:{username}` |
| —（新增） | `export_csv(default_name, content)` | 原生另存为对话框 + 写文件 |

请求头细节（`http.rs` 统一处理）：
- 所有请求：`X-Requested-With: XMLHttpRequest`；
- POST：`Content-Type: application/x-www-form-urlencoded; charset=UTF-8`、`X-CSRFToken`（从该环境 Jar 的 csrftoken 读）、`Origin: {origin}`、`Referer: {origin}/sqlquery/`（登录时为 `/login/`）；
- 会话失效检测：响应非 JSON 且形如 HTML → 返回「未登录或会话已过期」错误，前端沿用现有自动重登流程；
- 多环境隔离：`HashMap<envId, Client>`，每个 Client 独立 `reqwest::cookie::Jar`，复现浏览器按域隔离 Cookie 的行为（同域名不同环境 id 也各自独立，比浏览器更严格，无副作用）。

### 4.3 前端改动（`extension/` → `desktop/src/`）

| 文件 | 改动 |
|---|---|
| `lib/api.js` | `chrome.runtime.sendMessage` → `invoke('command', {...})`；接口签名对 app.js 保持不变；`describe` 在前端分别解析 MySQL DDL 与 PostgreSQL 列元数据 |
| `lib/store.js` | `chrome.storage.local` → `tauri-plugin-store`；`getCred/saveCred/clearCredPassword` 改调凭据 command（本地 JSON 只存 `{user, remember}` 标志，密码进凭据管理器） |
| `lib/crypto.js` | **删除**（被 DPAPI 取代） |
| `lib/actions.js` | 常量映射为 command 名，或直接内联进 api.js 后删除 |
| `app.js` | 保留业务编排，将资源树、表详情、事件、网格、SQL 编辑器和数据库上下文拆到 `lib/`；CSV 导出由 Blob 下载改为 `invoke('export_csv')` |
| `index.html/app.css` | 从扩展入口迁移并同步环境管理、多结果页签、PostgreSQL schema 等界面能力 |

### 4.4 删除项（桌面端不再需要）

- `manifest.json`、DNR 规则同步（`syncDnrRules`）、`chrome.cookies` 读取、`openApp` 标签页管理、`crypto.js`。

## 5. 执行计划（里程碑）

| # | 里程碑 | 内容 | 验收标准 |
|---|--------|------|----------|
| M1 | 工程脚手架 | 初始化 Tauri 2 工程；前端文件平移进 `desktop/src/`；窗口能打开显示现有 UI（数据为空态） | `cargo tauri dev` 打开主界面 |
| M2 | 网络层 + 登录 | `http.rs`（每环境 Client/Jar）+ `login/check_session` command；api.js 换 invoke | 测试环境能登录并保持会话 |
| M3 | 只读浏览链路 | `get_instances/get_resource/describe_table` + 前端表结构解析 | 树浏览、数据翻页、排序筛选、结构视图可用；MySQL 展示 DDL，PostgreSQL 不展示 DDL 页签 |
| M4 | 控制台 + 历史 + 存储 | `run_query`；store.js 迁移 plugin-store；查询历史、环境管理可用 | 控制台执行 SQL、历史留存、环境增删改 |
| M5 | 凭据 + 桌面特性 | keyring 凭据 command；记住密码/自动登录；CSV 原生另存为；窗口状态记忆 | 重启应用后记住密码的环境自动登录；导出弹原生对话框 |
| M6 | 打包验收 | `cargo tauri build` 单 exe；三环境全功能回归；文档更新 | 便携 exe 在干净机器（有 WebView2）上可用 |

依赖关系：M2 是关键路径（登录/CSRF/Cookie 是全部接口的前提）；M3/M4 可在 M2 后并行推进。

## 6. 风险与注意事项

1. **Django CSRF 严格校验**：若目标 dbadmin 开启 `CSRF_TRUSTED_ORIGINS` 白名单校验 Referer 的 https 场景，需保证 `Origin`/`Referer` 与目标完全一致（含端口）。M2 用真实测试环境尽早验证。
2. **WebView2 Runtime 依赖**：Win10/11 基本自带；打包配置保留 downloadBootstrapper 选项以防老机器。
3. **便携版配置目录**：实际使用 `%APPDATA%\com.fanxiaofan.sql-studio\store.json`，不跟随 exe；换机后 Windows 凭据不会随配置文件迁移，需要重新登录。
4. **凭据管理器边界**：DPAPI 防离线拷文件与跨用户，防不住同一 Windows 账户下的恶意进程——这是桌面端合理上限，已与用户确认接受。
5. **内网代理/自签证书**：当前实现使用 `no_proxy()` 强制直连，并无条件启用 `danger_accept_invalid_certs(true)`。这是受控内网兼容行为，不是公网安全默认值；当前尚未提供按环境关闭/开启的证书开关。
6. **双端行为一致**：桌面端与扩展端的共享模块由单元测试做逐字节一致性校验；宿主 API、存储和 CSV 导出保留各自实现。

## 7. 实施记录（2026-07-09 完成 M1–M6）

实际落地与方案的差异及过程中发现的问题：

1. **系统代理劫持（关键修复）**：开发机常见挂本地代理（如 `HTTP_PROXY=127.0.0.1:7897`），reqwest 默认读取环境变量代理，内网 dbadmin 域名走代理必然不通。已在 Client 构建时加 `no_proxy()` 强制直连；方案原第 5 条风险预判「默认走系统代理」正确但方向相反——内网场景必须**不走**代理。
2. **窗口入口文件名**：Tauri 窗口默认加载 `index.html`，扩展入口为 `app.html`，直接平移导致产物白屏「asset not found」。已将入口改名 `desktop/src/index.html`。
3. **凭据命名**：keyring v3 在 Windows 凭据管理器中的条目为 `LegacyGeneric:target={envId}.sql-studio`。
4. **CSV 表头修复**：扩展版 `exportCsv` 表头对列对象直接 `String()` 会输出 `[object Object]`（潜在 bug），桌面版已改为取 `c.name`。
5. **e2e 验证方案**（`desktop/test/`）：采用 `mock-dbadmin.js`（复现 Django csrftoken/sessionid/未登录回 HTML 语义）+ `e2e.js`（WebView2 CDP + playwright-core 驱动真实 exe）。测试覆盖登录、CSRF/Cookie、MySQL 三级树、PostgreSQL 四级 schema 树、数据/结构/DDL、多 SQL、历史、跨 schema 隔离和密码不落盘。CDP 连接必须设置 `NO_PROXY=127.0.0.1`；测试构建须使用 `test/tauri.e2e.conf.json` 注入调试参数，生产构建不开放调试端口。
6. **构建命令**：开发 `npm run dev`；生产 `npm run build`（= `tauri build --no-bundle`），产物 `desktop/src-tauri/target/release/sql-studio.exe`（约 10.4MB）；e2e 为 `node test/mock-dbadmin.js` + `node test/e2e.js <exe>`（需先用 e2e 配置构建）。
7. **默认环境同步**：扩展侧后来把内置环境抽为根目录 `default-envs.json`（仓库内为脱敏示例值），桌面版已同步该机制——`desktop/src/default-envs.json` + store.js 的 `loadDefaultEnvs()`，仅在本地无配置时作为初始值。
8. **PostgreSQL 模式层级**：当实例 `db_type=pgsql` 时，资源树与控制台使用“实例 → 数据库 → 模式 → 表”上下文；资源、表结构和查询接口均透传 `schema_name`。表数据浏览 SQL 使用 `"schema"."table"` 和 `OFFSET`，页大小通过 `/query/` 的 `limit_num` 交给 dbadmin 追加，避免客户端与服务端生成两个 `LIMIT`。
9. **前端模块化**：桌面端与扩展端新增 `sql-editor.mjs`、`db-context.mjs`、`resource-tree-view.mjs`、`table-view.mjs`、`app-events.mjs` 等共享模块；两端 `app.js` 均控制在 1000 行以内，共享模块由 parity 单测守护。
